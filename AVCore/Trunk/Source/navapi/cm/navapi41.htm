-          NAVAPI is a component.

-          It is built on multiple platforms:  Intel, DEC Alpha, and Solaris. 

-          It is released into the Perforce depot under a “Latest” directory and then later QA’d versions are released into “Release” directories.

-          It has 1 navapi.inf file and 2 client specs.

 

I.                  Requirements

 

-          Santa Monica Build system

-          Windows 95 or 98 – so you can build the dvx and nlm platforms.

-          Solaris Ultra 5 with Solaris 2.6, and GCC compiler #2.95.2 (compiler only) _____________

 

II.               Increment Build Number

 

1.        Open a 4dos window on your Windows 95/98 machine.

 

2.        Use your generic Perforce client to sync out the following file:

 

//depot/norton_antivirus/avcore/trunk/english/navapi/src/napiver.h

 

Mark the file for edit:

 

p4 edit napiver.h

 

3.        Modify the file and bump the build number.

 

4.        Submit back to Perforce.

IV.             p4bldit

 

Note:

 

Navapi has one inf file (navapi.inf) and 2 client specs:

 

                WIN32:                  navapi41_win32_cm_client

                SOLARIS:              navapi41_sol_cm_client

 

                The client spec name for WIN32 is specified in the navapi.inf file.

                The stamp for WIN32 is specified in the navapi.inf file as well:

 

                                navapi41_win32_cm.##

 

-          Grab the navapi.inf file from //depot/norton_antivirus/avcore/trunk/source/navapi/cm and copy to your local c:\build directory.

-          Modify  - change all “.xx” to the current NAVAPI build number.

-          Run p4bldit against the navapi.inf file as follows:

 

p4bldit navapi.inf /l

 

 

 

 

 

V.                Set up the directories

 

  1. Run the dirs batch files out of your c:\build\hold directory:

navcoredirs

navavcoredirs

 

VI.             Sync the Intel client navapi_win32_cm_client

 

  1. Run the c:\build\hold\navsync.bat file to sync files and sync the label:

navsync

Note:  The S: drive will be mapped to:  s: \\ussm-dev1\vol4\bld_area\navapi.##

Note:  The label will be:  navapi_cm.##

 

VII.          Build NAVAPI for Intel and DEC Alpha

 

  1. Build NAVAPI by running the batch files:

navcorebld

navavcorebld

On both an Intel machine and a DEC Alpha machine.

Note:      For Intel, build on Windows 95/98 only.  There are many platforms (wra, dvx, nlm) that

                don’t build on NT.

                For Alpha, build these on the DEC Alpha (obviously!)

  1. Check for errors.  You can ignore the following broken projects:

NAVAPIST.QA.DVX

NAVAPIST.QA.NLM

 

 

VIII.       Sync the Solaris client cm_navapi_unix

 

Manually sync the Solaris client using the same label as Intel as follows:

  1. After verifying it has been backed up, delete the previous build of NAVAPI as follows:

a.    cd /export/home/cmsource/navapi

b.       rm -rf *   (Be very careful when using this command that you are

in the right location.)

Manually sync the Solaris client using the same label as Intel as follows:

  1. P4CLIENT=navapi41_sol_cm_client
  2. export P4CLIENT

 

NOTE:  Show the client spec listing here:

 

  1. p4 sync –f

Note:  The build environment will be synced to /export/home/cmsource/navapi

  1. p4 label navapi41_sol_cm.##
  2. p4 labelsync –l navapi41_sol_cm.##
  3. Lock the label.

IX.             Build NAVAPI on Solaris

 

  1. cd /export/home/cmsource/navapi
  2. chmod 775 tolower
  3. chmod 775 makeall
  4. tolower     This changes all file names in the current tree to have lower case letters.
  5. makeall release
  6. makeall debug

 

The following files should be produced in your bin.release and bin.debug directories:

 

apitest

libnavapi.a

libaveng.a

libsymkrnl.a

 

Back on Intel:

 

  1. Create the following directory:

 

s:\navapi\latest.dev\libsolaris

 

  1. Using Winftp or Samba, copy the files just built (above) from the bin.release directory to the same directory:

 

s:\navapi\latest.dev\libsolaris

 

 

X.                Post the build into Latest (automate this)

 

  1. Map the s: drive letter as follows:

net use s: /d

net use s: \\ussm-dev1\vol4\bld_area\navapi.##\avcore\source

At this level, from the root of the S: drive, the directory “navapi” will be visible.

  1. At the command line, set your client to the one that checks into the release area:

set p4client=release_navapi41_win32_cm_client

The client looks like this:

Client:       release_navapi41_win32_cm_client

…

Root:         s:\NAVAPI\LATEST.DEV

 

Options:     noallwrite noclobber nocompress crlf unlocked modtime

 

View:

                //depot/release/navapi/version4/latest/... //release_navapi/...

 

  1. Flush your client.  You do NOT want to sync because that would write over the current build sitting on your S: drive!

p4 flush -f

4.     cd navapi\latest.dev

5.     dir /s /b /a:-d | p4 -x - edit

6.     p4 submit

 

 

If you are building Solaris only, do this:

 

  1. Set your client spec:

P4CLIENT=release_navapi41_solarisonly_cm_client

export P4CLIENT

  1. Flush your client:

p4 flush -f

  1. Perform an add & edit operation:

find . –type f –print | p4 –x – add

find . –type f –print | p4 –x - edit

  1. Submit the files to the depot:

p4 submit

 

 

XI.             Release the binaries as requested (automate this)

 

  1. When the team requests that a certain platform be released, you need to copy the files from the requested build into the Release area.  The Release area has the following structure:

 

//depot/Release/navapi/version4/releases/libsolaris/release.001/include/navapi.h

   (Comes from \bld_area\navapi.##\avcore\source\navapi\latest.dev\include)

//depot/Release/navapi/version4/releases/libsolaris/release.001/doc/navapi.doc

   (Comes from \bld_area\navapi.##\avcore\source\navapi\latest.dev\docs)

//depot/Release/navapi/version4/releases/libsolaris/release.001/binary1…

//depot/Release/navapi/version4/releases/libsolaris/release.001/binary2…

      (These come from the appropriate directories in

 \bld_area\navapi.##\avcore\source\navapi\latest.dev\…)

 

 

Instead of doing the above, just make a branch:

 

1.        Open the Windows Perforce GUI.

2.        Open your generic client spec that allows you to see the whole depot.

3.        Right click on the latest folder you are branching:

//depot/Release/navapi/version41/latest/

4.        From the Pop-up menu, choose “Integrate” and then “using Filespec”.

5.        In the dialog box that appears, in the Target Filespec, type the name of the destination folder:

//depot/Release/navapi/version41/release.00#/...

6.        Click “Options”.  Check “Do not copy new files to workspace”.  This will prevent

Files from being copied to your local machine, which greatly speeds up the process.

Click  “OK” to return to the previous dialog box.

7.        When ready, click “OK”.  You will see the message

“Opened ## files for integrate.”

8.        Go to your “Submitted changelists view” (2 triangles).  Right click on your Default changelist and select “submit”.

9.        When done, notify the team.

 

 

XII.          Generate Metrics

 

  1. Run the following command:

 

metrics //depot/… navapi.pp navapi.##

 

where     pp = the previous build number

                and         ## = the current build number

 

                For example, for build # 39 of Navapi:

 

                metrics //depot/… navapi.38 navapi.39

 

                Post the files into Perforce somewhere.

 

 

XIII.       Inform the dev team that the build is done.

 

XIV.       Zip up the Build.

 

XV.          Enter your build notes into the Access build database.

 

 

 

 

 

 

Notes for Partial Builds

 

A.     Things to remember about doing a touch-up for NAVAPI in AVCORE:

 

-          Rename the navapi.## build area to the next name, such as “38A”.

-          Often the NAVAPI team likes to do a partial but treat the numbering as though it was a full build.  This is acceptable.  Put this information in the CM build database.

-          Use the navapi41_win32_cm_client client.

-          Set your s: drive to the following mapping:

 

net use s: \\ussm-dev1\vol4\bld_area\navapi.##

 

This will match the client.

 

-          Sync the files requested in the partial instructions.

-          Create a label and labelsync it:  Example:  navapi41_win32_cm.##

 

B.      Use “env” plus the batch file “navadrv” to set up your environment;  proj and build! 

For example:

 

        env avcore

        navadrv 14

        proj navapi ira

        m -a

 

                                The CORE that is used is part of the navapi41 build environment.  We do not use

                                the pre-built core builds on dev1\vol3.

 

 

 

 

 

You are now done.
